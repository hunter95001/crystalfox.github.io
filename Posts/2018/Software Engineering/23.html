요구분석

요구분석은 무엇을 개발할지 결정합니다.

요구분석 단계

1.도메인 분석
-문제를 해결하기에 앞서 문제 자체를 이해하는 작업

2.요구추출

3.분석 및 명세화
-어떤 기능을 가져야 하는지 정확히 기술하는 단계

4.검토 및 수정

도메인 분석

개발하려는 분야의 지식을 알아가는 과정이다.
고객이 일하는 분야의 비즈니스나 기술을 의미한다.
빠른 개발과 좋은 시스템을 만들 수 있는 장점이다.

요구추출

해결하려는 문제를 정의하고 프로젝트 범위를 결정한 후 여러 가지 방법을 활용하여 요구를 추출한다.

기능적 요구 -외형적으로 나타내는 기능이나 성능등의 동작

- 시스템이 무엇을 하는지, 사용자나 다른 시스템을 위해 제공하는 서비스가 무엇인지 기술한 것이다.
컴퓨팅 –계산이 일어남.

비기능적 요구 –기능이 아닌 성능, 효율, 반응시간, 제약 조건 등

소프트웨어를 개발하는 동안 고수해야할 제약 조건으로 하드웨어 자원의 제약 소프트웨어 품질의 특성 수준의 범위를 정해놓은 것이다.

요구 추출

요구 추출 방법 작업 방법 특징
관찰 사용자의 업무를 관찰 하여 기록 감추어진 문제를 잘 드러냄
인터뷰 여러 관련 당사자를 만나 질문하고 대답을 기록 정확한 요구 추출이 가능하며 요구에 대한 오해를 줄임
브레인스토밍 여러 사람이 모인 그룹에서 아이디어를 쏟아 놓음 효과적인 정보 추출이 가능함
프로토타이핑 시범적으로 시스템을 구현 요구에 대한 빠른 피드백이 가능함
사용 사례 분석 시스템의 외부 기능 파악 체계적인 요구 구성이 가능함

관찰

사용자의 업무를 관찰하는 것이야말로 대화에서 놓치기 쉬운 자세한 사항을 파악할 수 있는 방법이다.
관찰하고 결과 정보를 분석하는데에는 시간이 많이 소요된다.

인터뷰

인터뷰하는 사람은 무엇보다도 고객과 사용자에게 공감하고 잘 듣는 훈련이 되어 있어야 한다. 경청과 공감.

브레인 스토밍

회의하는 방법

프로토타이핑

프로토타이핑의 목적은 소프트웨어 엔지니어의 아이디어에 대한 피드백을 조기에 받아서 요구 사항을 취합하는 것

사용자 사례분석

개발한 소프트웨어를 가지고 사용자가 무엇을 할 수 잇는지를 검증하는 체계적인 접근 방법이다.

사용 사례 간 관계

확장 관계
-사용 사례의 처음부터 끝까지 모든 단계를 기술하고 예외 상황을 다루는 부분을 포함해야 한다는 것이다.

상세화 관계
-일반화된 사용사례와 유사사용 사례의 관계를 나타낸다.

포함 관계
-여러 사용 사례 사이에 공통적인 부분이 있을 때 이를 사용 사례의 일부로 분리하여 나타낼 수 있다. 여러 사용 사례가 일련의 액션을 공유하는 것.
