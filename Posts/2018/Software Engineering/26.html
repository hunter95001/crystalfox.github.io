디자인 패턴 –대표적으로 사용하는 것을 일반화함.
비슷한 역할의 클래스 유형을 자주 쓰게 된다.
자주 반복되는 설계 유형을 디자인 패턴이라 한다.
디자인 패턴은 체계적으로 문서화하여 범용적으로 사용할 수 있도록 정리한 것이다.

가장 널리 알려진 디자인 패턴은 Gang of Four의 23가지 패턴이다.

싱글턴 패턴 사용
시스템의 형상 정보는 한 객체에만 존재하게 하여 전역 변수처럼 접근해야 한다.
시스템 로그 파일이나 라이브러리 카탈로그는 시스템에 하나만 존재해야 한다.
패턴이 자세한 내용은 클래스 다이어그램이나 시퀸스 다이어그램으로 표현한다
클래스 다이어그램에는 참여 객체와 그것들의 역할 및 의무, 관계를 나타내고,
시퀸스 다이어그램에는 참여 객체가 문제를 해결하기 위해 어떻게 상호작용 하는지 나타낸다

적용되지 않은 경우 –반복, 중복되는 코드가 많아 재사용이 어려움
디자인 패턴 명세 – 이름, 배경, 문제, 솔루션, 혜택과 책임, 가이드라인, 관련 패턴


개념 실체 패턴 – 각 객체의 공통 정보를 공유할 때 사용
실체 – 공통된 정보를 가진 맴버들의 모임
개념 – 공유하는 정보를 담는 클래스

플레이어 역할 패턴 – 하나의 클래스에 다양한 역할을 표현하고 싶을 때 사용한다
역할이 다양한 것을 표현하기 위해 플레이어 객체와 속성의 집합으로 나누는 것

위임 패턴 – 다른 클래스가 가진 특정 오퍼레이션을 활용하여 책임을 위임하고 싶을대 사용
다른 클래스의 오퍼레이션에 작업을 요청하는 것
코드중복 피함
근접한 정보에만 접근

계층 구조 패턴 – 계층 관계를 가진 객체들을 묶어 동일한 처리를 하고 싶을 때.
트리 구조를 사용한다.
계층 구조에 잇는 각 객체는 하부 층에 여러 객체를 종속시킬 수 있다.
